## Why Manual Proposal Creation Is Costing You More Than You Think

Let's do the honest maths.

A single proposal takes 45 minutes to build manually. You send five proposals a month. That's **3 hours and 45 minutes** — every single month — not thinking, not strategising, not doing anything creative. Just moving information from one place to another.

And that's when everything goes right. When it goes wrong? Wrong client name on the cover page. Last month's pricing still in the budget section. A scope section you forgot to update. These aren't careless mistakes — they're *inevitable* ones when you're assembling documents by hand under pressure.

The real cost isn't just time. It's the first impression. A proposal with errors signals disorder before the project has even started.

The fix is straightforward: build the proposal once as a template, and let an automation fill it in every time.

## How It Works: Step by Step

### Step 1: Build Your Proposal Template in Google Docs

Open Google Docs and build your standard proposal layout. This should be your *best* proposal — the one you'd want every client to receive. Include all the sections you typically include:

- Introduction / context
- Scope of work
- Deliverables
- Timeline
- Investment / pricing
- Terms and next steps

Wherever client-specific information goes, replace it with a **placeholder variable** in double curly braces. These are the spots the automation will fill in automatically.

```
&#123;&#123;client_name&#125;&#125;
&#123;&#123;company_name&#125;&#125;
&#123;&#123;project_scope&#125;&#125;
&#123;&#123;budget&#125;&#125;
&#123;&#123;start_date&#125;&#125;
&#123;&#123;delivery_date&#125;&#125;
&#123;&#123;your_name&#125;&#125;
```

> 📸 *Screenshot: Google Doc proposal template with placeholder variables highlighted in yellow*

> 💡 **Pro tip:** Keep your variables consistent. If you use `&#123;&#123;client_name&#125;&#125;` in one place, don't use `&#123;&#123;Client Name&#125;&#125;` in another — the automation is case-sensitive. Write them down in a list so you remember them all.

Once your template is saved, note the **File ID** from the Google Docs URL — you'll need it in Make.com later.

> ✅ **After this step:** You have one master proposal template with clearly marked placeholders. You'll never edit this document directly again — only the variables change.

---

### Step 2: Create Your Intake Form in Tally

Now build the form that captures everything your proposal needs. Use **Tally** (free and excellent) or Typeform.

Your form fields should map **directly** to the variables in your template. One variable = one form field.

| Template Variable | Form Field Label |
|---|---|
| `&#123;&#123;client_name&#125;&#125;` | Client's Full Name |
| `&#123;&#123;company_name&#125;&#125;` | Company / Organisation |
| `&#123;&#123;project_scope&#125;&#125;` | What does this project involve? |
| `&#123;&#123;budget&#125;&#125;` | Project Budget |
| `&#123;&#123;start_date&#125;&#125;` | Desired Start Date |
| `&#123;&#123;delivery_date&#125;&#125;` | Expected Delivery Date |

Keep the form focused. Only ask for what the *document* needs. You can ask additional discovery questions separately.

> 📸 *Screenshot: Tally form showing the fields above, with field labels visible*

> 💡 **Pro tip:** In Tally, you can use hidden fields to pre-populate things like today's date or your own name — useful for variables like `&#123;&#123;proposal_date&#125;&#125;` and `&#123;&#123;your_name&#125;&#125;` so you don't have to type them every time.

> ✅ **After this step:** You have a form that, when submitted, contains all the information needed to build a complete proposal.

---

### Step 3: Set Up the Automation in Make.com

This is the part where the magic happens. You'll build a **scenario** in Make.com — a chain of connected modules that each do one thing. Here's what your scenario needs to do:

**Module 1 — Tally: Watch for New Submissions**
This is your trigger. Make.com watches your Tally form and fires the scenario every time someone submits it.

**Module 2 — Google Drive: Copy a File**
Make.com creates a fresh copy of your proposal template in Google Drive. This is important — you never edit the original template, only copies.

Set the destination folder to wherever you store client proposals. Name the copy something like: `Proposal — &#123;&#123;client_name&#125;&#125; — &#123;&#123;today's date&#125;&#125;`.

**Module 3 — Google Docs: Replace Text in a Document**
This is where the placeholders get swapped out. For each variable in your template, map the corresponding form field response.

Make.com will go through the document and replace every instance of `&#123;&#123;client_name&#125;&#125;` with the actual name from the form, `&#123;&#123;budget&#125;&#125;` with the actual budget, and so on.

**Module 4 — Gmail (Optional): Send the Document**
If you want the proposal delivered automatically — to yourself for review, or directly to the client — add a Gmail module here. Attach the newly created Google Doc as a PDF and send.

> 📸 *Screenshot: Make.com scenario showing all four modules connected in sequence*

> 📸 *Screenshot: Close-up of the Google Docs "Replace Text" module with variable mappings*

> 💡 **Pro tip:** Use Make.com's "Convert to PDF" module before the Gmail step if you want to send a PDF rather than a Google Doc link. Clients are more likely to open a PDF attachment.

> ✅ **After this step:** Your automation is live. Every form submission now produces a finished proposal in your Google Drive within seconds.

---

### Step 4: Add a Review Step (Recommended for the First Month)

Speed is great — but accuracy matters more. Before your proposal goes to a client automatically, you probably want to check it first.

Add an intermediate step: instead of emailing the client directly, have Make.com send the document **to you** first, with a short note: *"New proposal created for &#123;&#123;client_name&#125;&#125; — review here: [link]"*.

You review it. If it looks good, you forward it to the client manually. If something needs adjusting, you do it in the copy.

Once you've run this 10–15 times and you trust the output completely, you can remove the review step and let it go straight to the client.

## Common Mistakes to Avoid

**Not standardising your template before automating.**
If your proposals are all different shapes and sizes, automation will just make them consistently messy. Before you build any of this, spend an afternoon creating *one* proposal you're proud of. That's what you automate.

**Using too many variables.**
Every variable is a field on the form. The longer the form, the more friction there is to fill it in. Start with the essentials — name, company, scope, price, date. You can always add more later.

**Mismatched variable names.**
`&#123;&#123;client_name&#125;&#125;` and `&#123;&#123;Client_Name&#125;&#125;` are different things to Google Docs. Make sure every placeholder in your template matches exactly what you've mapped in Make.com.

**Skipping the review step on launch day.**
Run the automation at least five times with real data before trusting it completely. Check that the formatting looks right, all variables populated correctly, and nothing was skipped.

**Sending the wrong file to a client.**
Make sure the "Copy a File" module is set up correctly — it should copy the *template*, not overwrite it. Double-check the file IDs in Make.com.
